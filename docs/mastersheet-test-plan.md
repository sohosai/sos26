## 事前準備データ

- **企画**: P1, P2, P3（最低3つ。FがP1のowner）
- **フォーム**: Aが作成。TEXT / NUMBER / SELECT / CHECKBOX / FILE の FormItem を含む
- **フォーム配信**: Bが承認。P1・P2に配信済み、P3は未配信
- **フォーム回答**: P1は提出済み、P2は下書き保存済み

## 3. FORM_ITEMカラム（セル状態表示）

> 前提: Aがフォーム作成・配信承認済み。P1=提出済み、P2=下書き、P3=未配信。

| # | テスト | 対象企画 | 期待状態 | 期待表示 |
|---|--------|----------|----------|----------|
| 3-1 | 未配信 | P3 | NOT_DELIVERED | 「─」（グレー）、編集不可 |
| 3-2 | 配信済み・未回答 | 新たにP3に配信後 | NOT_ANSWERED | 「─」、編集不可 |
| 3-3 | 下書き保存済み | P2 | NOT_ANSWERED | 「─」、編集不可（下書き回答値は非表示） |
| 3-4 | 提出済み | P1 | SUBMITTED | 値表示（通常色） |
| 3-5 | 実委が編集済み | P1（Aが編集後） | COMMITTEE_EDITED | 編集値が表示 |
| 3-6 | 再提出による状態リセット | P1（企画側再提出後） | SUBMITTED | 再提出値が表示。実委の編集は履歴に残るが表示値は再提出値 |

### 3-6 の再現手順

1. Aがフォーム配信済み → P1（F）が回答を提出
2. AがマスターシートでP1のセルを編集 → COMMITTEE_EDITED確認
3. F（P1のowner）がフォーム回答を**再提出**
4. Aがマスターシートを**リロード** → **SUBMITTED**に変化。再提出の値が表示されることを確認

## 推奨テスト実行順序

```
1. アカウント・データ準備
   └─ A〜Fのアカウント作成、Bureau・権限設定
   └─ 企画P1〜P3作成、フォーム作成・配信・回答

2. カラム作成テスト（§1）
   └─ 各タイプのCUSTOMカラム + FORM_ITEMカラム作成
   └─ 編集・削除の権限確認

3. カラム可視性テスト（§2）★重点
   └─ PRIVATE / PUBLIC(ALL / BUREAU / INDIVIDUAL) の表示確認
   └─ FORM_ITEMカラムのフォーム権限連動
   └─ viewer変更による表示切替

4. セル状態テスト（§3）
   └─ 4状態の遷移確認（NOT_DELIVERED → NOT_ANSWERED → SUBMITTED ↔ COMMITTEE_EDITED）
   └─ 再提出による状態リセットの確認

5. セル編集テスト（§4, §5）
   └─ CUSTOMカラムの各データ型編集
   └─ FORM_ITEMカラムの編集 + 読取専用型の確認
   └─ 再提出による編集の上書き確認

6. 閲覧申請テスト（§6）★重点
   └─ 申請→承認/却下フロー
   └─ FORM_ITEMカラムのFormCollaborator自動作成

7. ビュー・フィルター（§7, §8）
   └─ ビューCRUD + 状態永続化
   └─ 各フィルター型の動作確認

8. 表示値の統一テスト（§9）★重点
   └─ マスターシート・委員会回答画面・企画回答画面で同じ値が表示されることを確認
   └─ FormItemEditHistoryのフォールバック動作

9. エッジケース（§10）
   └─ 競合・大量データ・CSV・URL共有
```

> **★重点テスト**: §2（可視性）、§6（閲覧申請）、§9（表示値の統一）は権限モデルとデータ整合性に直結するため、最も丁寧にテストすること。
